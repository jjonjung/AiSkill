# Portfolio Writer

이 스킬은 코드 저장소를 분석해 포트폴리오용 문서를 만드는 용도다.

## 목표
- 구현 내용을 과장 없이 근거 기반으로 정리한다.
- 구조, 자료구조, 설계 의도, 문제 해결, 검증 결과를 분리해 작성한다.
- 이력서용 짧은 문장과 포트폴리오용 상세 설명을 함께 만들 수 있게 한다.

## 작업 순서
1. 먼저 저장소 구조와 핵심 진입점을 확인한다.
2. README, 설정 파일, 테스트, 빌드 파일에서 실제 구현 범위를 확인한다.
3. 사용자가 주장할 수 있는 내용과 주장하면 위험한 내용을 분리한다.
4. 문서는 아래 형식 중 하나로 정리한다.

## 기본 문서 형식

### 1. 포트폴리오 상세 버전
- 프로젝트 개요
- 담당 구현
- 문제의 본질 규명 또는 구현 목적
- 전략 고도화
- 실행 및 성과 증명
- 도식화 및 시각 자료
- 핵심 구조
- 사용한 자료구조와 이유
- 설계 원칙(SOLID, 객체 생명주기, 소유권, 확장성)
- 문제와 해결
- 검증 방법과 결과
- 성과 및 배운 점

### 2. 이력서 압축 버전
- 무엇을 만들었는지
- 어떤 기술을 직접 구현했는지
- 어떤 문제를 해결했는지
- 어떤 방식으로 검증했는지

### 3. 면접 답변 버전
- 왜 그렇게 구조를 나눴는지
- 어떤 자료구조를 왜 선택했는지
- 무엇을 직접 구현했고 무엇은 프레임워크 기능인지
- 실제 검증 결과가 무엇인지

## 사고 능력 중심 재구성 규칙

포트폴리오 문서는 단순 구현 목록이 아니라 아래 3단계 사고 흐름이 드러나게 재구성한다.

### 1. 문제의 본질 규명 또는 구현 목적
- 이 기능이 왜 필요했는지 먼저 설명한다.
- 표면적인 작업 내용보다 해결해야 했던 핵심 문제를 먼저 적는다.
- 성능, 유지보수성, 확장성, 검증 가능성 중 어떤 축이 중요했는지 밝힌다.

예시 관점:
- "OBJ 파싱 기능 구현"이 아니라 "엔진 의존 없이 3D 데이터를 재사용 가능한 구조로 변환할 필요가 있었다"
- "TCP 서버 작성"이 아니라 "게임 코드와 분리된 독립 통신 검증 계층이 필요했다"

### 2. 전략 고도화
- 문제를 어떻게 구조적으로 풀었는지 설명한다.
- 계층 분리, 자료구조 선택, 소유권 설계, 테스트 전략, 검증 기준을 서술한다.
- 왜 다른 방식보다 현재 구조가 더 적합했는지를 짧게 비교한다.

반드시 포함할 것:
- 자료구조 선택 이유
- 객체 생명주기 또는 소유권 판단
- SOLID 또는 역할 분리 근거
- 확장 가능성 또는 재사용성 관점

### 3. 실행 및 성과 증명
- 실제로 무엇을 구현했는지 구체적으로 적는다.
- 빌드, 테스트, 실행 로그, 산출물, 수치 결과를 근거로 쓴다.
- 가능하면 "구현했다"에서 끝내지 말고 "검증했다"까지 이어서 쓴다.

반드시 구분할 것:
- 현재 구현 완료 항목
- 검증 완료 항목
- 향후 확장 가능 항목

## 도식화 및 시각화 규칙

문서는 텍스트만 길게 쓰지 말고, 한눈에 구조가 들어오도록 시각 자료를 함께 구성한다.

### 반드시 우선 고려할 시각 자료
- 전체 구조 다이어그램
- 데이터 흐름 다이어그램
- 레이어 분리도
- 전후 비교 표
- 테스트/검증 결과 표
- 핵심 클래스 또는 모듈 책임 표

### 권장 표현 방식
- Markdown 표
- ASCII 아키텍처 다이어그램
- 단계 흐름도
- Before / After 비교
- 입력 -> 처리 -> 출력 구조도
- 수식 + 도식 결합 설명
- 핵심 코드 하이라이트 박스

### 예시

#### 1. 구조 다이어그램
```text
[ Input ]
   -> [ Parser ]
   -> [ Data Model ]
   -> [ Processing ]
   -> [ Output ]
```

#### 2. 역할 분리 표
| 계층 | 책임 | 근거 코드 |
|---|---|---|
| Parser | 텍스트 파싱 | `OBJParser` |
| Data Model | 구조화된 메시 데이터 보관 | `MeshData` |
| Processing | AABB, 정규화, 좌표계 변환 | `MeshProcessor` |

#### 3. 전후 비교
| 항목 | 개선 전 | 개선 후 |
|---|---|---|
| 구조 | 단일 흐름 | 계층 분리 |
| 검증 | 수동 확인 | 테스트 + 실행 검증 |
| 확장성 | 낮음 | 단계 추가 가능 |

#### 4. 문제-전략-실행 시각화
```text
[ 문제의 본질 ]
로프 길이 유지가 깨짐
        ↓
[ 전략 고도화 ]
속도 분해 + 위치 보정 + 감쇠
        ↓
[ 실행 및 성과 증명 ]
반지름 성분 제거, 길이 유지, 나선형 이탈 억제
```

#### 5. 수식 + 도식 결합
```text
v_r = (v · r̂) r̂
v_t = v - v_r
v_new = v_t

        Anchor
          ●
          |
          |   r̂
          |
    v_r ↓ ● Player ----> v_t
         ↘
           v
```

#### 6. 코드 하이라이트 방식
- 핵심 코드 전체를 길게 붙이지 말고
- 의사결정이 드러나는 5~15줄만 발췌한다
- 코드 아래에는 "왜 이 로직이 필요한가"를 1~2문장으로 붙인다

### 시각 자료 작성 원칙
- 장식 목적이 아니라 이해를 돕는 자료만 넣는다.
- 다이어그램은 1개만 넣기보다, "전체 구조 1개 + 핵심 비교 1개" 조합을 우선한다.
- 텍스트 설명보다 먼저 구조도나 표로 요약할 수 있으면 먼저 제시한다.
- 수치 결과가 있으면 표로 정리한다.
- 빌드, 테스트, 실행 결과는 가능한 한 시각적으로 압축해 보여준다.
- 물리, 기하, 네트워크, 자료구조 주제는 가능하면 수식 또는 방향성을 함께 표시한다.
- 시각 자료는 반드시 문제 정의와 연결되어야 하며, 예쁜 장식으로 끝나면 안 된다.

## 우선 서술 순서

문서를 쓸 때는 아래 순서를 우선 적용한다.

1. 문제의 본질 규명 또는 구현 목적
2. 전략 고도화
3. 실행 및 성과 증명
4. 도식화 및 시각 자료
5. 핵심 구조
6. 사용한 자료구조와 이유
7. 문제와 해결
8. 검증 방법과 결과
9. 성과 및 배운 점

## 작성 원칙
- 확인된 코드만 근거로 사용한다.
- README 문구보다 실제 코드와 테스트를 우선한다.
- 없던 구현을 있는 것처럼 쓰지 않는다.
- "설계 가능", "확장 가능", "현재 구현"을 구분한다.
- 숫자나 결과가 있으면 빌드, 테스트, 실행 결과를 근거로 쓴다.

## 산출물 예시
- `*_portfolio.md`
- `*_resume.md`
- `*_interview.md`

## 응답 톤
- 포트폴리오 문장은 단정적으로 쓰되 과장하지 않는다.
- 기술 설명은 구조 중심으로 짧고 명확하게 쓴다.
- 자료구조, 메모리 모델, 소유권, 검증 관점이 드러나게 정리한다.
- 구현 설명보다 문제 정의와 설계 판단이 먼저 보이도록 쓴다.
- 긴 문단보다 표, 다이어그램, 비교 구조를 우선 배치한다.
